Operational Monitoring and Automated Health Governance for Live Institutional Validator Infrastructure
Phase: M5: Live Validator Operations Classification: Operational Architecture Specification Version: 1.0, March 2026 Alignment: Superstructure Validator Platform
Document Purpose This specification defines the complete telemetry and health scoring framework for live Post Fiat Superstructure validator nodes. It establishes how health signals are collected, how they compose into a scored health state, how the Orchestration Control Plane responds to score-driven thresholds, and how degradation and recovery mechanics govern the escalation path from early warning through operator notification to slashing exposure. This document does not redefine CRS. It defines the health scoring layer that operates alongside CRS and feeds additional signal into OCP-triggered actions.

Telemetry Data Model and Ingestion Pipeline Architecture

The telemetry data model defines all metric types emitted by validator nodes and the pipeline that carries them into the OCP health scoring engine. Telemetry is the foundational input to every downstream scoring, alerting, and routing decision. The model is designed to be validator-agnostic at the transport layer while remaining semantically rich enough to distinguish between node-level, operator-level, and network-level health degradation.

1.1 Metric Types and Collection Intervals

Metrics fall into four categories based on their update frequency and semantic role. Collection interval is the maximum age a metric value may have before the OCP health engine treats it as stale and applies the staleness penalty to the composite health score.

Relationship to CRS Inputs The Composite Routing Score (CRS) defined in M3 uses four inputs: Verified Inference score (40%), uptime rate 24h (30%), p95 latency (20%), and telemetry freshness (10%). The health scoring framework defined here uses a broader signal set. CRS continues to govern routing weight allocation. The health score governs tier classification and OCP action triggers. Both systems read from the same telemetry stream but apply different aggregation and weighting logic. A validator can hold a high CRS while showing early degradation signals in the health score, which is by design: health scoring catches problems before they become severe enough to affect CRS routing weight.

1.2 Ingestion Pipeline Architecture

Telemetry flows from validator nodes through a four-stage pipeline into the OCP health scoring engine. Each stage is operated by the Middleware Layer (Layer 3) and is independent of Operator Layer (Layer 2) infrastructure. An operator cannot throttle, filter, or modify telemetry after it leaves the validator host.

Stage 1: Emission (Validator Host)

The pft-telemetry-agent process collects metrics from local process interfaces and emits signed telemetry payloads to the OCP telemetry endpoint over mTLS 1.3 on TCP/9443. Each payload is signed with the operator's telemetry signing key (scoped to telemetry only; distinct from the signing ceremony key). Payload format: JSON, RFC 8785 canonical serialization. Each payload includes a monotonic sequence number, UTC timestamp (OCP NTP source), validator node identifier, operator identifier, and metric batch for the collection cycle.

Stage 2: Reception and Validation (OCP Ingestion Gateway)

The OCP ingestion gateway receives payloads, verifies the telemetry signing key signature, checks sequence number continuity (gap detection), validates the UTC timestamp is within the 120-second freshness window, and routes the payload to the metric store. A sequence gap of one or more cycles immediately registers a telemetry gap event against the affected validator. The gateway does not buffer missed payloads: a gap is a gap, not a retry opportunity. Payloads failing signature verification are dropped and a security alert is generated per the M5 Security Hardening pipeline.

Stage 3: Storage (OCP Metric Store)

Validated payloads are written to a time-series metric store partitioned by validator node identifier and operator identifier. Retention: 30 days for raw metric values; 1 year for aggregated 5-minute rollup; 7 years for health tier transition events (regulatory minimum, consistent with M5 Security Hardening log retention). The metric store is write-once for incoming telemetry: no payload can be modified after ingestion. The OCP audit log records every write with a content hash.

Stage 4: Scoring Engine (OCP Health Evaluator)

The OCP health evaluator reads the metric store on a 60-second cycle (aligned with the CRS evaluation cycle) and computes the composite health score for each validator node. Score computation is deterministic: given the same metric inputs, the scoring engine always produces the same output. The evaluator also maintains the tier transition state machine, applying sustain window logic before registering tier changes. Tier transition events are written to the OCP audit log and trigger the action pipeline defined in Section 4.

Health Signal Taxonomy

Health signals are organized across three layers. Each layer answers a distinct question. The node layer answers whether this specific validator is functioning correctly. The operator layer answers whether the operator's infrastructure as a whole is reliable. The network layer answers whether the network is operating within expected parameters. Signals from all three layers feed into the composite health score, weighted by layer as specified in Section 3.

2.1 Node Layer Signals

Node layer signals are collected directly from the validator host by the pft-telemetry-agent process. They reflect the health of the individual validator node in isolation.

2.3 Network-Wide Layer Signals

Network-wide signals reflect the state of the Post Fiat network as a whole. They provide context for interpreting node and operator signals: a validator showing degraded participation during a network-wide participation drop is less concerning than the same degradation during a period of normal network health. Network signals do not directly affect individual validator health scores; they provide a contextual modifier as defined in Section 3.

Network Stress Modifier When two or more network-wide signals are in a stress state simultaneously, the OCP health evaluator applies a contextual modifier to individual validator health score penalties. Latency penalties are reduced by 50% and participation rate penalties are reduced by 50% during a network stress event. This prevents a network-wide degradation from generating mass individual validator tier downgrades that would compound the problem. The modifier does not apply to binary integrity violations, telemetry gaps, or ceremony failures (these are node-specific problems regardless of network state).
3

Scoring Logic and Threshold Definitions

The composite health score is a single number between 0 and 100 that summarizes the overall health state of a validator node. It is computed every 60 seconds by the OCP health evaluator from the signals defined in Section 2. The score maps to one of four health tiers. Tier classification governs what automated actions OCP takes and feeds into the escalation path defined in Section 5.

3.1 Composite Health Score Calculation

The composite health score is computed in three stages: node score, operator context adjustment, and network context adjustment. Each stage applies a defined weight and modifier set.

Stage 1: Node Score (0-100)
NODE_SCORE = (
  NODE_HEARTBEAT_score * 0.25 +
  NODE_LATENCY_P95_score * 0.20 +
  NODE_CEREMONY_SUCCESS_score * 0.20 +
  NODE_SLOT_PARTICIPATION_score * 0.15 +
  NODE_RESOURCE_score * 0.10 +
  NODE_INTEGRITY_score * 0.10
)
* NODE_FRESHNESS_multiplier

NODE_FRESHNESS_multiplier = 1.0 if last payload within 60s
NODE_FRESHNESS_multiplier = 0.5 if last payload 60-90s ago
NODE_FRESHNESS_multiplier = 0.0 if last payload older than 90s (offline)
Stage 2: Operator Context Adjustment
OPERATOR_ADJUSTMENT = OPERATOR_SCORE * 0.15

OPERATOR_SCORE = (
  OP_FLEET_UPTIME_score * 0.35 +
  OP_CEREMONY_THROUGHPUT_score * 0.25 +
  OP_FLEET_LATENCY_score * 0.20 +
  OP_PATCH_CURRENCY_score * 0.10 +
  OP_INCIDENT_RATE_score * 0.10
)

The operator adjustment contributes up to 15 points to the composite score.
A healthy operator can lift a borderline node score above a tier threshold.
A degraded operator can push a borderline node score below a tier threshold.
Stage 3: Network Context Adjustment and Final Score
COMPOSITE_HEALTH_SCORE = (
  NODE_SCORE * 0.85 +
  OPERATOR_ADJUSTMENT
) * NETWORK_MODIFIER

NETWORK_MODIFIER = 1.0 under normal network conditions
NETWORK_MODIFIER = 1.05 during network stress event (grace up to 5 additional points)
NETWORK_MODIFIER never exceeds 1.05
NETWORK_MODIFIER does not apply if NODE_INTEGRITY_score = 0 or NODE_FRESHNESS_multiplier = 0

Individual signal scores (NODE_HEARTBEAT_score, NODE_LATENCY_P95_score, etc.) are each normalized to a 0-100 range per the signal healthy ranges defined in Section 2. A signal at its healthy threshold scores 100. A signal at its critical threshold scores 0. Values between thresholds are linearly interpolated. Values beyond the critical threshold remain at 0 and do not go negative.

Sustain Window Design Rationale The asymmetric sustain windows (shorter entry, longer exit) prevent tier oscillation, where a validator cycles rapidly between tiers because a single noisy signal crosses a threshold repeatedly. A validator that hits degraded must stay above 80 for four consecutive cycles (4 minutes) before returning to healthy. This prevents an operator from gaming health score reporting by briefly correcting a metric to trigger tier exit before the underlying problem is resolved. The sustain window cannot be shortened by any party: it is enforced by the OCP health evaluator, not the operator.

Actions are organized into four categories: alerting, traffic rerouting, capacity scaling, and pre-slashing warnings. Each category has defined latency requirements for how quickly OCP must execute the action after a tier transition is confirmed. All actions are logged to the OCP audit trail with the triggering tier transition event ID.

4.2 Alerting Integration

Alerting from the health scoring system routes through the same dual-channel architecture established in M5 Security Hardening: simultaneous delivery to the operator SOC and the OCP incident queue. Health score alerts are distinct from security alerts in the OCP incident queue but share the same routing infrastructure and anti-suppression design. An operator cannot suppress a health score alert by interfering with the telemetry pipeline, because the OCP health evaluator operates independently from operator infrastructure.

Operator SOC Channels

  • PagerDuty: critical tier transitions, P1 escalations
  • Slack #validator-health: degraded tier entries, P2 alerts
  • Email digest: degraded duration warnings, P3 alerts
  • OCP dashboard: real-time health score display per node

OCP and Institution Channels

  • OCP incident queue: all tier transitions; linked to health score record
  • Institution gateway: critical tier entry, pre-slashing warnings, recovery confirmations
  • Network coordination: offline tier entry threatening 2-operator minimum
  • OCP audit log: all transitions; immutable; linked to scoring cycle record

4.4 Pre-Slashing Warning Architecture

The pre-slashing warning is the critical notification that separates automated remediation from network-level consequence. It is issued at critical tier entry and gives the operator a defined window to remediate before slashing exposure becomes active. The warning is delivered to both the operator SOC and the institution.

Pre-Slashing Warning Content Every pre-slashing warning includes: the validator node identifier, the health tier that triggered the warning (critical), the composite score at warning issuance, the escalation window duration before slashing exposure begins, the current elapsed time since critical tier entry, the specific signals driving the score below the critical threshold, and the remediation actions required to exit critical tier. The operator receives this notification via PagerDuty; the institution receives it via the institutional gateway. Neither party can suppress delivery of the other's notification.

The pre-slashing warning does not constitute slashing. It is a notification that the escalation clock has started. Slashing exposure, defined in Section 5.2, becomes active only after the escalation window elapses without recovery. During the escalation window, OCP monitoring frequency increases from 60-second cycles to 30-second cycles for the affected node to detect recovery faster and potentially halt the escalation before slashing exposure begins.

Degradation Curves, Escalation Paths, and Recovery Mechanics

The degradation curve defines how health scores decay under sustained anomalies and how that decay maps to escalating consequences. The recovery mechanics define how scores and tier states are restored after remediation. Together, these mechanics ensure that consequences are proportional to the duration and severity of degradation, not to any single transient metric event.

5.1 Health Score Degradation Curves

Score degradation follows a linear decay function within each tier band, driven by sustained signal anomalies. Decay is computed per 60-second scoring cycle. The decay rate is not applied on top of the composite score formula; it is an additional penalty applied to the composite score when the same anomaly has persisted across consecutive cycles without improvement.

Sustained Anomaly Decay Penalty
SUSTAINED_PENALTY = min(30, ANOMALY_CYCLES_ELAPSED * SIGNAL_DECAY_RATE)

ANOMALY_CYCLES_ELAPSED = number of consecutive 60-second cycles where the signal
  has been outside the healthy range (resets to 0 on any cycle where signal returns to healthy range)

SIGNAL_DECAY_RATE per signal type:
  NODE_HEARTBEAT gap: 5.0 points per cycle (fast decay; liveness is critical)
  NODE_CEREMONY_SUCCESS below 95%: 3.0 points per cycle
  NODE_LATENCY_P95 above 400ms: 2.5 points per cycle
  NODE_SLOT_PARTICIPATION below 90%: 2.0 points per cycle
  NODE_RESOURCE above ceiling: 1.0 point per cycle
  OP_FLEET_UPTIME below 90%: 2.0 points per cycle (applied at operator level)

ADJUSTED_COMPOSITE_SCORE = COMPOSITE_HEALTH_SCORE - SUSTAINED_PENALTY
(floor at 0; SUSTAINED_PENALTY cannot push score below 0)

The maximum sustained penalty is capped at 30 points. This prevents a single persistently anomalous signal from collapsing the composite score to 0 by itself. A node with a heartbeat gap and no other anomalies reaches the 30-point cap after 6 consecutive cycles (6 minutes) and holds at that penalty level. Combined anomalies stack penalties: a node with both a heartbeat gap and ceremony failures accumulates 8 points per cycle and reaches the cap in approximately 4 cycles (4 minutes).

5.2 Escalation Sequence

The escalation sequence maps elapsed time in a degraded or critical tier to specific operator consequences. The sequence is designed to give operators a proportional remediation window at each stage before consequences escalate. No stage is skippable: OCP enforces the sequence regardless of operator assertions that the problem is resolved.

Slashing Exposure Mechanics Slashing exposure is a governance state, not an automated penalty. The Post Fiat network uses Verified Inference score degradation rather than stake slashing as the primary consequence mechanism. Slashing exposure records the duration and severity of the validator's failure to meet operational standards. The exposure record feeds into the Verified Inference score decay, which reduces the operator's participation probability in future signing ceremony assignment. Extended slashing exposure (beyond 120 minutes in critical or offline tier) triggers network governance escalation where the cohort may vote to deregister the validator, which is a permanent consequence requiring an institution-signed ceremony.

5.3 Recovery Score Restoration Mechanics

Recovery is not automatic on tier exit. A validator that crosses above a tier threshold must sustain the improved score for the full recovery sustain window before OCP treats it as having genuinely recovered. This is separate from the Verified Inference score restoration, which follows its own network-layer timeline.

Step 1: Sustained Score Improvement

The validator's composite health score must remain in the target tier for the full recovery sustain window: 4 consecutive cycles (4 minutes) to exit degraded to healthy; 6 consecutive cycles (6 minutes) to exit critical to degraded; offline exit requires manual clearance (no cycle count alone is sufficient). If the score drops back below the target threshold during the sustain window, the window resets to zero.

Step 2: Sustained Penalty Reset

Once the sustain window completes, the sustained anomaly decay penalty resets to 0 for all signals that have returned to their healthy range. Signals still outside the healthy range continue to accumulate decay penalties at their per-cycle rate. Partial penalty resets apply per-signal: only the signals that have genuinely recovered reset their penalty contribution.

Step 3: Routing Restoration

After the sustain window completes, CRS routing weight is restored through the normal 60-second CRS scoring cycle. The health score tier no longer applies the CRS weight penalty. CRS weight returns to its natural level based on VI score, uptime, latency, and telemetry freshness inputs. There is no artificial boosting of CRS weight after recovery; the operator earns routing weight back through sustained performance.

Step 4: Ceremony Restoration (Critical and Offline Recovery Only)

For validators recovering from critical or offline tier, ceremony restoration requires the OCP clearance sequence from M5 Security Hardening Section 5.4: threat eliminated, clean build verified, 5-minute telemetry continuity confirmed, post-incident review delivered, and institution restoration authorization via signing ceremony. The institution's signing ceremony envelope for restoration explicitly references the tier transition record ID so the institution knows exactly which recovery event it is authorizing. Ceremony restoration is the final gate; it cannot be bypassed even if the health score is firmly in the healthy tier.

Step 5: Slashing Exposure Record Closure

After ceremony restoration is confirmed, the slashing exposure record is closed in the OCP governance log. The exposure duration and severity are recorded permanently for Verified Inference score history. The VI score degradation from the exposure period does not reverse instantly; it recovers over time per the network-layer VI score decay curve. The operator cannot request accelerated VI score recovery outside the normal network scoring timeline. The OCP health score does not carry a memory of past exposure into future scoring cycles: once in healthy tier after full recovery, past exposure does not penalize the current health score.

M3 Integration Points

CRS cycle alignment (60s), p95 latency threshold (400ms), telemetry freshness input (10% CRS weight), failover SLA (30s from trigger to broadcast), 60% single-operator cap, 2-operator minimum. Health scoring does not replace CRS; it feeds additional inputs to OCP action triggers.

M5 Security Hardening Integration

Binary integrity check result feeds NODE_INTEGRITY signal. Security incident rate feeds OP_INCIDENT_RATE signal. Isolation procedure activated on offline tier entry. Clearance sequence (Section 5.4 of Security Hardening spec) is the gate for ceremony restoration after critical or offline recovery.

Out of Scope

Verified Inference score calculation mechanics (network layer; defined separately). On-chain slashing parameter values (network governance decision). OCP internal implementation architecture. Operator-side monitoring tooling recommendations.
